# UVM 学习笔记生成 Prompt

以下 prompt 用于生成或修改 UVM 学习笔记。使用时直接粘贴即可。

---

## 风格约束

1. 全文不使用"你"字，需要时用"验证工程师"替代。
2. 标题和正文不使用口语化表达。禁止以下句式："会说话了"、"最没存在感的"、"我是 xxx"、"壳子搭一次永远不动"。
3. 正文不用数学符号表达关系（如 `=`、`+`、`→`），用完整的句子描述。类树图放在代码块里时可以使用符号。
4. 正文段落不使用 bullet point（`-` 分点）罗列信息，整合成连贯的段落。表格和代码块可以正常使用。
5. 不使用夸张或煽动性语气（如"不改一行代码"、"一行搞定"、"干干净净"）。
6. 语言目标：清晰但不口语，专业但不生硬。允许使用比喻辅助理解，但不过度。

## 结构要求

每次讲解一个概念模块时，按以下顺序组织：

1. 先回答两个问题：**这一层加了什么**、**为什么需要单独分出这一层**。
2. 然后展开细节，包括代码示例（如涉及）。
3. 模块之间用"为什么分开"的逻辑串联，而不是罗列继承链。

类树图放在代码块里时可以使用 ASCII 符号，正文描述时不用符号表达关系。

## 内容范围

**这一节不是固定的——需要根据当前要生成的内容临时代入。** 只写本次要讲哪些话题、覆盖到什么程度。例如：

- 如果本次是生成类树全景笔记，写：transaction 支线从 uvm_transaction 讲到 sequencer；component 支线从 uvm_report_object 讲到 phase。
- 如果本次是专门讲 factory，写：factory 的注册-创建-替换三个环节，用对比示例说明为什么要这个机制，并展开相关语法（宏、`::`、`type_id`、`create()`）。
- 如果本次是专门讲 phase，写：9 个 phase 的完整列表、执行顺序、function phase 与 task phase 的区别、`raise_objection` 的作用。

**原则：内容范围是"本次笔记要覆盖什么"，每次根据当前对话内容填写，不保留上一次的范围。**

---

## 使用示例

```
请用以下约束生成/修改 UVM 学习笔记：

[风格约束 — 固定不变]
[结构要求 — 固定不变]
[内容范围 — 本次临时填写，如：详细解释 factory 机制的注册-创建-替换流程，以及涉及的语法（宏展开、:: 作用域解析、type_id、create()）]
```
